### 第 1 题 — create_agent vs create_deep_agent

**难度:** 基础
**题目:** 在 LangChain 中，`create_agent` 和 `create_deep_agent` 有什么区别？什么场景下应该选择 `create_deep_agent`？

**我的回答:**
> create_agent 适用于 ReAct 模式，create_deep_agent 适用于 harness 模式。create_deep_agent 自带很多 middleware，如果做复杂场景或者长任务选择 create_deep_agent，如果是工作流和简单场景选择 create_agent。

**评分: 6/10**

**优点:**
- 正确指出 create_deep_agent 自带更多 middleware/内置工具，适合复杂场景
- 正确判断 create_agent 适用于简单场景

**不足:**
- "ReAct 模式" vs "harness 模式"不是主流 LangChain 社区术语，会让面试官困惑
- 未说明两者来自不同库：create_agent 来自 `langchain.agents`（官方），create_deep_agent 来自 `deepagents`（第三方包）
- 未提到 create_deep_agent 注入大量内置工具导致 token 消耗大的坑
- 未展开说明 create_deep_agent 额外能力（context_schema、ToolRuntime、子代理编排）

**标准回答:**
> `create_agent` 是 LangChain 官方的标准 Agent 创建方式，基于 ReAct 模式，支持 tools、system_prompt、response_format 和 checkpointer，适合常见的工具调用和结构化输出场景。
>
> `create_deep_agent` 来自 `deepagents` 第三方库，是 LangChain 的扩展。它额外支持 context_schema/ToolRuntime 实现类型安全的业务上下文注入、内置大量文件系统/执行工具，以及子代理编排能力。适合需要复杂上下文传递、多步编排的长时间任务。但它的内置工具会显著增加 token 消耗，不适合简单问答场景。

### 第 2 题 — Context vs thread_id vs config

**难度:** 基础
**题目:** 请解释 LangChain Agent 中的 context 是什么？它和 thread_id / config 有什么区别？什么情况下传了 context 但实际不会被用到？

**我的回答:**
> context 是在单次会话中存放业务数据的上下文。thread_id 是可以跨会话的，而 context 是用于单次会话。如果是两次会话都是同一个 thread_id，同一个问题，LLM 可能不走 tool。

**评分: 7/10**

**优点:**
- 正确区分了 context（单次业务数据）和 thread_id（跨会话记忆标识）
- 准确提到了"同一个 thread_id + 同一个问题 → LLM 可能不走 tool"这个关键坑点

**不足:**
- 回答过于碎片化，没有形成完整的技术解释结构
- 缺少 config 的对比（config 是运行时配置容器，thread_id 只是 config.configurable 下的一个字段）
- "thread_id 是可以跨会话的"表述不够准确
- 未说明 context 需要自己拼接到 system_prompt 中才能被 LLM 感知

**标准回答:**
> **context** 是每次 `agent.invoke()` 时传入的类型化业务参数（如 user_id、is_vip），通过 `ToolRuntime[Context]` 在 tool 函数中读取。它绑定到 **单次 invoke**，不会跨调用传递。
>
> **thread_id** 是 `config.configurable` 下的一个字段，用于对话记忆。同一个 thread_id 的多次 invoke 共享消息历史（通过 checkpointer 保存），LLM 可以看到上下文轮次。
>
> **config** 是 LangChain 的运行时配置容器，包含 thread_id、callbacks、metadata 等。
>
> **context 不会生效的场景：** 同一个 thread_id 下，LLM 可能从历史记忆中直接回答，不需要调 tool，此时即使传了 context，tool 没执行，context 就不会被用到。解决方案：不同问题用不同 thread_id，或在 system_prompt 中强制调 tool。

### 第 3 题 — stream_events vs invoke

**难度:** 中等
**题目:** stream_events 和 invoke 的区别？什么时候用 stream？stream 返回的消息有哪些类型？

**我的回答:**
> stream_events 和 invoke 都是基于事件触发器模式，stream_events 是每次触发了某个事件（如 AIMessage）就会输出，invoke 要等到所有触发器事件都执行完成才输出。当需要给用户展示消息或者调试时使用。消息类型有 AIMessage、tool_call。

**评分: 5/10**

**优点:**
- 正确指出 invoke 是"等全部完成再返回"，stream_events 是"每步都推"
- 提到了 AIMessage 和 tool_calls

**不足:**
- "都是基于事件触发器模式"概念混淆，invoke 是同步阻塞调用，stream_events 是迭代器/generator
- 使用场景说得太笼统，核心场景是"多 tool 调用时展示进度"
- 消息类型不全：除了 AIMessage，还有 HumanMessage、ToolMessage（共三种）
- 未提到 version="v3"的实验性协议特点和 LangChainBetaWarning

**标准回答:**
> `invoke` 是同步调用，等待所有步骤（LLM 推理→调 tool→观察结果→再推理→最终回复）完成后一次性返回最终结果。用户全程看不到中间过程。
>
> `stream_events` 返回一个迭代器，**每执行一步就 yield 一次当前的完整 state**（v3 协议）。调用方可以在循环中实时获取进度：比如看到 LLM 调了什么 tool、tool 返回了什么、最终回复是什么。适合耗时较长的多 tool 场景。
>
> 消息类型有三种：**HumanMessage**（用户输入）、**AIMessage**（AI 回复，含 **tool_calls** 或 **content**）、**ToolMessage**（tool 执行结果）。
>
> 注意 `version="v3"` 是实验性协议，每次返回完整 state 而非增量 diff，会打印 `LangChainBetaWarning`。

### 第 5 题 — 对话记忆持久化

**难度:** 中等
**题目:** LangChain 中对话记忆的实现方式有哪些？InMemorySaver 的局限性？生产环境怎么做？

**我的回答:**
> 忘了。InMemorySaver 只能在当前线程中使用，程序一旦重启就会把记忆丢失。需要自己使用自定义 memory，存放到文件或者数据库中。

**评分: 5/10**

**优点:**
- InMemorySaver 进程重启后丢失 ✅
- 生产环境需要持久化存储 ✅

**补充要点:**
- 其他实现方式：SqliteSaver、PostgresSaver、AsyncPostgresSaver、自定义 BaseCheckpointSaver
- InMemorySaver 另一个局限：多进程下无法跨实例共享
- 生产方案：单机用 SqliteSaver，分布式用 PostgresSaver 或 Redis/MongoDB

**标准回答:**
> LangChain 中对话记忆的实现方式：
> 1. **`InMemorySaver`** — 内存存储，仅限单进程，重启丢失
> 2. **`SqliteSaver` / `SqlSaver`** — SQLite/关系型数据库持久化
> 3. **`PostgresSaver` / `PostgresCheckpointer`** — PostgreSQL 存储，适用于生产
> 4. **自定义 Checkpointer** — 实现 `BaseCheckpointSaver` 接口，适配 Redis/MongoDB 等
>
> **`InMemorySaver` 的局限性：**
> - 进程重启 → 所有记忆丢失
> - **非进程间共享**：多 Agent Server 场景下，每个进程有自己的内存，无法跨实例同步
> - 无持久化能力
>
> **生产环境方案：**
> - 单机：`SqliteSaver`（轻量、零部署）
> - 分布式：`PostgresSaver` 或自定义 Redis/MongoDB 实现
> - 结合业务：将 thread_id 映射到用户会话 ID，实现用户级别的对话历史持久化

**评分: 4/10**

**优点:**

**不足:**
- batch 和 batch_as_completed 的核心区别没说对
- batch 是"等所有完成按顺序返回"，batch_as_completed 是"按完成顺序逐个返回"

### 第 7 题 — 限流设计

**难度:** 进阶
**题目:** 大模型应用中为什么需要限流？InMemoryRateLimiter 的使用场景和局限性？分布式环境怎么设计？

**我的回答:**
> 因为模型对调用频率有限制，或者出于成本考虑。适用于单机场景，如果是多个 agent、多个模型，无法做到整体限流。需要自己使用基于 Redis 的方案。

**评分: 7/10**

**优点:**
- 正确说明限流的两个核心原因：API 频率限制和成本控制 ✅
- 准确指出 InMemoryRateLimiter 仅限单机场景 ✅
- 提到分布式方案需要基于 Redis ✅

**补充要点:**
- InMemoryRateLimiter 的参数含义：requests_per_second、check_every_n_seconds、max_bucket_size（令牌桶算法）
- "无法做到整体限流"更准确说：多进程下每个进程各有计数器，无法跨进程感知
- 常见方案：Redis + 令牌桶/漏桶/滑动窗口算法

**标准回答:**
> **为什么需要限流：**
> - API 层面：模型供应商（如 DashScope、OpenAI）对调用频率有限制，超限会返回 429 或限流
> - 成本层面：控制 token 消耗，避免意外的高额账单
>
> **`InMemoryRateLimiter` 的使用场景和局限性：**
> 你的 `qwen_model.py` 配置了 `requests_per_second=1`、`check_every_n_seconds=0.1`、`max_bucket_size=10`，这是令牌桶算法的三个核心参数。它**仅限单机/单进程**场景——多进程部署下，每个进程有各自的计数器，A 进程发了 100 个请求不会被 B 进程的限流器感知到，从而突破整体上限。
>
> **分布式限流方案：**
> - Redis + 令牌桶算法（最常用，通过 Lua 脚本保证原子性）
> - Redis + 滑动窗口算法（更精确控制时间窗口）
> - 分层设计：API 级别限流（全局）+ 用户级别限流（每个 API Key 独立配额）

### 第 8 题 — init_chat_model vs ChatQwen

**难度:** 进阶
**题目:** init_chat_model（OpenAI 兼容模式）和 ChatQwen 有什么区别？什么场景下必须用 ChatQwen？

**我的回答:**
> 忘了。ChatQwen 在 reasoning 有自己特别的参数。如果使用的模型是 DashScope 提供的模型，最好使用 ChatQwen。

**评分: 4/10**

**优点:**
- 知道 ChatQwen 处理 reasoning 有独特优势 ✅
- 知道 DashScope 模型用 ChatQwen 更合适 ✅

**补充要点:**
- `init_chat_model`（OpenAI 兼容模式）：DashScope 的 `reasoning_content` 字段被 OpenAI 适配器丢弃，拿不到思考过程
- `ChatQwen`（langchain_qwq 包）：DashScope 原生协议，正确解析 `reasoning_content`
- 项目中的命名混乱问题：`deepseek_model.py` 实际初始化的是 `glm-5.1`（智谱）

**标准回答:**
> `init_chat_model(..., model_provider="openai")` 走的是 **OpenAI 兼容模式**。它通过 LangChain 的 OpenAI 适配器调用 DashScope 的 OpenAI 兼容接口。问题是：DashScope 返回的 `reasoning_content` 字段（思考链内容）在 OpenAI 协议中没有对应字段，**会被 LangChain 适配器丢弃**，所以即使 `enable_thinking=True`，也拿不到思考过程。
>
> `ChatQwen` 来自 `langchain_qwq` 第三方包，走的是 **DashScope 原生协议**，能正确解析 `reasoning_content`。用法：
> ```python
> model = ChatQwen(model="qwen3.7-plus")
> for chunk in model.stream([...]):
>     print(chunk.additional_kwargs.get('reasoning_content', ''))
> ```
>
> 所以如果你的模型是 DashScope 提供的，且需要 reasoning 能力，**必须用 `ChatQwen`**。
>
> 另外项目的命名有混淆：`deepseek_model.py` 实际初始化的是 `glm-5.1`（智谱的模型），`thinking_model.py` 初始化的是 `qwen3.7-plus`。面试时主动指出这一点，说明你有代码审查意识。

### 第 9 题 — Tool 注册方式 + ToolRuntime

**难度:** 进阶
**题目:** LangChain Agent 中 Tool 的注册方式有哪些？内置工具、自定义函数、@tool 装饰器各有什么优缺点？ToolRuntime 和普通 @tool 的区别？

**我的回答:**
> 一个在 create_agent 注册，还有就是 model.bind_tools；内置工具更好的使用模型的能力，自定义函数工具可以实现一些自定义的功能，@tool 装饰器不知道；ToolRuntime 也不知道区别是什么。

**评分: 3/10**

**优点:**
- 提到了两种注册方式（create_agent 传入、model.bind_tools）

**不足:**
- `model.bind_tools` 不是注册到 Agent，而是绑定到 LLM 模型自身
- `@tool` 装饰器已在项目中导入（`agent_stream.py:36`）但未答出
- ToolRuntime 的作用完全没答出来

**标准回答:**
> **Tool 的三种注册方式：**
> 1. **直接传 function**（`tools=[get_weather]`）— 最简单，LangChain 自动用 @tool 包装。你项目大多数文件用这种方式。
> 2. **`@tool` 装饰器** — 显式从 `langchain.tools` 导入，可以为 tool 指定 `name`、`description`、`args_schema`，更精细控制。你项目 `agent_stream.py:36` 和 `agent_context.py:35` 已导入 `from langchain.tools import tool`。
> 3. **`model.bind_tools(tools)`** — 绑定到 LLM 模型自身，让模型知道可用工具但不经过 Agent 编排。适合自己实现 ReAct 循环的定制场景。
>
> **`ToolRuntime[T]` 的作用：**
> - **没有 ToolRuntime 时**：tool 函数只接收 LLM 传入的参数，拿不到外部业务数据
> - **有 ToolRuntime[T] 时**：tool 函数多声明一个 `runtime: ToolRuntime[Context]` 参数，通过 `runtime.context.field` 拿到 `invoke()` 时传入的业务上下文（如 user_id、is_vip）
> - **核心价值**：实现了**业务参数的类型化注入**，避免在 tool 参数中透传无关数据

>                 cls._instances[model_id] = ChatQwen(...)
